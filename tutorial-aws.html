01What Is an Attack Surface?

// every way in is a risk

An attack surface is the total set of points where an unauthorized person could try to enter or extract data from a system. Think of it like the exterior of a building: every door, window, ventilation shaft, and skylight is a potential entry point. The more entry points you have, the harder it is to secure them all.

In cybersecurity, reducing the attack surface is a core principle. If a service doesn't need to be public, make it private. If an IAM role doesn't need admin access, scope it down. Every unnecessary exposure is a door left open.

Exposed vs. Hardened Surface
Left: a system with many exposed entry points (doors open). Right: the same system after hardening — most access points are closed, leaving only what's necessary.

02Why Cloud Infrastructure Is a Target

// complexity breeds misconfiguration

AWS environments can contain hundreds of interconnected services — compute instances, databases, storage buckets, API endpoints, identity roles, and networking layers. This complexity makes it remarkably easy to leave something misconfigured.

Common real-world entry points include:

• Public S3 buckets — storage exposed to the entire internet, sometimes containing sensitive data
• Exposed API Gateways — endpoints without proper authentication or rate limiting
• Overpermissioned IAM roles — identity roles with wildcard ("do anything") policies
• Unpatched EC2 instances — compute servers running outdated software with known vulnerabilities

03How This Simulation Models an AWS Environment

// procedural generation of a fake cloud

When the page loads, the simulation procedurally generates a fake but plausible AWS infrastructure. It picks a mix of service types (EC2, S3, RDS, Lambda, IAM, VPC, CloudFront, etc.), assigns each one randomized attributes, and wires them together with edges.

Edges between nodes represent three kinds of relationships:

• Network connectivity — EC2 talks to RDS over a private subnet
• Trust relationships — an IAM role is assumed by an EC2 instance
• Data flow paths — CloudFront routes traffic to an EC2 origin, which queries RDS

The graph starts with a spanning tree (ensuring every node is reachable from every other) and then adds extra edges based on the edge density parameter. More edges mean more potential lateral movement paths for attackers.

05The Risk Score Formula

// how three factors combine into one number

Each node's risk score (0–100) is a weighted sum of three factors. The weights reflect a common real-world prioritization: vulnerability severity matters most, but exposure and asset value together can outweigh a critical bug hiding behind a firewall.

Risk Score Formula
Risk = Severity × 0.40 + Exposure × 0.35 + Value × 0.25
Each factor is normalized to 0.0–1.0 before weighting. The result is scaled to 0–100. Patched nodes have their score reduced by the patch effectiveness multiplier.
Pseudocode: Risk Score Calculation
function compute_risk(node):
  severity_score = severity_to_number(node.vuln_severity) / 4 -- None=0, Low=1, Med=2, High=3, Crit=4
  exposure_score = exposure_to_number(node.exposure) / 2 -- Internal=0, Public=1, Internet=2
  value_score = value_to_number(node.asset_value) / 3 -- Low=0, Med=1, High=2, Crit=3

  raw = severity_score * 40 + exposure_score * 35 + value_score * 25

  if node.is_patched:
    raw = raw * (1 - PATCH_EFFECTIVENESS)

  return min(raw, 100)
Why weighting matters: An S3 bucket with a critical vulnerability AND internet exposure scores higher than a database with a critical vulnerability but internal-only access. Exposure amplifies severity because it determines who can actually reach the vulnerability.

06How Threats Spawn & Spread

// smart attackers pick the best targets

Threats enter the simulation at internet-exposed or public-facing nodes — the services that are reachable from outside the network. These are the same entry points real attackers scan for: public S3 buckets, open API Gateways, CloudFront distributions, and IAM endpoints.

Once a threat compromises a node, it doesn't spread randomly. It evaluates connected neighbors and prioritizes the highest-value, most-exposed targets first. This simulates how sophisticated attackers (and automated attack tools) work — they don't wander aimlessly, they head straight for the crown jewels.

Pseudocode: Threat Spreading Algorithm
-- Called when a node becomes COMPROMISED
function spread_threat(compromised_node):
  neighbors = get_connected_nodes(compromised_node)

  -- Filter to uncompromised, unpatched neighbors
  targets = neighbors where phase == "safe" and not patched

  -- Sort by attractiveness to attacker (highest first)
  sort targets by (exposure * 2 + asset_value * 3) descending

  -- Spread to top targets based on aggression
  spread_count = ceil(targets.length * AGGRESSION)
  for each target in targets[0..spread_count]:
    if random() < AGGRESSION * 0.8:
      target.phase = "recon"
      log("LATERAL MOVEMENT initiated")
Design choice: The sort-by-attractiveness step is what makes the simulation feel "intelligent." Without it, threats would spread uniformly and the attack would look like a random flood. With it, you can watch threats make beeline paths toward databases and IAM roles.

07Attack Chain Phases

// recon → exploit → compromise → lateral movement

Each node under attack progresses through four distinct phases. The transition speed depends on the threat aggression parameter — higher aggression means fewer ticks between phases.

1. Reconnaissance
The attacker probes the node — scanning ports, checking versions, enumerating endpoints. No damage yet, but the clock is ticking.
Visual: subtle pulsing glow on the node outline.
2. Exploit
A vulnerability is being actively leveraged. The attacker has found a way in and is attempting to gain access or execute code.
Visual: node color intensifies toward amber/red.
3. Compromised
The node is fully taken over. The attacker has control and can exfiltrate data, modify configurations, or use this node as a staging point.
Visual: peak heat color, persistent glow, animated edges to neighbors.
4. Lateral Movement
From the compromised node, the attacker pivots to connected systems — beginning recon on the next highest-value target.
Visual: animated red dots travel along edges to neighbor nodes.

08The Heat Map Layer: Visualizing Risk

// color tells the story

Each node in the graph is colored according to its current risk score. This is the "heat map" layer — it turns abstract numbers into an immediately readable visual signal. Cooler colors mean safer nodes; hotter colors mean higher risk.

Compromised nodes are visually distinct from merely high-risk nodes: they carry a persistent glow and their edges to neighbors animate with red pulses showing active lateral movement. This distinction is important — a node can have a high risk score (it's vulnerable and exposed) without actually being under attack yet.

09Walking Through a Simulated Attack

// a realistic scenario, start to finish

Let's trace a realistic attack path through the simulation. This is a common pattern in real-world cloud breaches.

Tick 0–3: The simulation spawns a threat at cf-public-cdn-04, a CloudFront distribution with internet exposure. The attacker begins reconnaissance — scanning the CDN's configuration for misrouted paths or exposed origin servers.

Tick 4–7: Recon completes. The threat enters the exploit phase, leveraging a misconfigured origin fetch path (CVE-2024-4102) to access the backend EC2 instance ec2-prod-web-server-12 directly.

Tick 8–11: The EC2 instance is compromised. The attacker now has shell access and scans the local network. Three neighbors are visible: an RDS database (Critical value), an SQS queue (Low value), and another EC2 instance (Medium value). The threat prioritizes the RDS database.

Tick 12–18: Lateral movement begins. Recon starts on rds-prod-database-01. Meanwhile, the alert log fills with timestamped events tracking every phase transition. If the user patches the EC2 instance now, the lateral movement is halted — but the CloudFront entry point remains compromised.

Pseudocode: Full Per-Tick Threat Engine
function simulation_tick():
  -- 1. Spawn new threats at exposed entry points
  exposed = nodes where phase == "safe" and exposure in ["Internet", "Public"]
  repeat SPAWN_RATE times:
    if random() < 0.3 * AGGRESSION:
      pick random node from exposed
      set phase = "recon"

  -- 2. Progress existing threats through phases
  for each node where phase != "safe":
    ticks_in_phase = current_tick - node.phase_start_tick
    phase_duration = 3 + (1 - AGGRESSION) * 8 -- faster at high aggression

    if phase == "recon" and ticks_in_phase >= phase_duration:
      set phase = "exploit"
    else if phase == "exploit" and ticks_in_phase >= phase_duration:
      set phase = "compromised"
      spread_threat(node) -- lateral movement begins

  -- 3. Update risk scores (threat phase boosts score)
  for each node:
    base = compute_risk(node)
    if phase == "recon": base = max(base, 30)
    if phase == "exploit": base = max(base, 60)
    if phase == "compromised": base = max(base, 90)
    node.risk_score = base

  -- 4. Save snapshot for timeline replay
  save_snapshot()
Design choice: Active threats boost the minimum risk score of a node regardless of its base attributes. This ensures that even a low-value, internal node glows hot when it's actively compromised — because in reality, any compromised node is a serious problem.

10What the Parameters Actually Do

// a reference for every control
Parameter Controls Low Extreme High Extreme
Threat Aggression Speed of threat phase transitions and lateral movement probability 0.0: threats barely progress; very slow, cautious attacker. 1.0: near-instant phase transitions; aggressive, automated attack.
Spawn Rate New threat entry points per tick 1: slow drip of one new probe at a time. 10: coordinated multi-vector attack, many entry points simultaneously.
Patch Effectiveness Risk score reduction multiplier when a node is patched 0.0: patches do nothing (worst case — patching fails). 1.0: patches eliminate all risk (idealized instant remediation).
Asset Count Total number of AWS service nodes generated 20: small environment, easy to track visually. 80: complex enterprise-scale environment, harder to defend.
Edge Density Number of connections between nodes beyond the minimum spanning tree 0.1: sparse, isolated services. Limits lateral movement paths. 1.0: dense mesh. Many lateral movement paths available to attackers.
Tick Speed Simulation ticks per second 1 TPS: slow motion, study each phase transition. 20 TPS: fast-forward through the attack timeline.

11Real-World Parallels & Limitations

// what's realistic and what's simplified

What this simulation gets right:

• Attack surfaces grow with infrastructure complexity
• Threats enter at exposed endpoints and move laterally
• Attackers prioritize high-value, high-exposure targets
• Risk is a composite of vulnerability severity, exposure, and asset value
• Patching and hardening reduce risk but may not eliminate it

What this simulation simplifies or omits:

• Real attacks involve credential theft, privilege escalation, persistence mechanisms, and many more techniques
• Real infrastructure has layered defenses (firewalls, WAFs, IDS/IPS, MFA) that actively block attacks
• Real attackers use tools like Metasploit, Cobalt Strike, or custom malware — not a simple phase state machine
• Real vulnerability scoring uses CVSS (Common Vulnerability Scoring System), a much more granular framework
• The procedurally generated graph does not model real AWS networking (VPC peering, security groups, NACLs)

Real tools that do similar work:

• AWS Security Hub — aggregates security findings from AWS services
• Amazon GuardDuty — ML-based threat detection for AWS accounts
• Tenable and Qualys — vulnerability scanning and attack surface management
• MITRE ATT&CK — framework mapping real-world attacker techniques

Important caveat: This is a visual learning tool, not a real vulnerability scanner. It generates fake data and simulates attack patterns to build intuition about cloud security concepts. It cannot detect real vulnerabilities, and no real infrastructure is being scanned or tested.

13Inside the Code — Graph Model

// how the simulation represents an AWS environment

The environment is a directed graph: two flat arrays hold everything. nodes[] stores each service's position, type, and three risk attributes (severity, exposure, asset value); edges[] stores source/target index pairs. An adjacency map is built once at generation so neighbor lookups are O(1) instead of scanning every edge each tick.

Core Data Structures
const nodes = []; -- { id, x, y, type, severity, exposure, value, compromised, score }
const edges = []; -- { from, to }
const adj = new Map(); -- id -> [neighborId, ...]

function buildAdjacency() {
  adj.clear();
  for (const e of edges) {
    (adj.get(e.from) || adj.set(e.from, []).get(e.from)).push(e.to);
    (adj.get(e.to) || adj.set(e.to, []).get(e.to)).push(e.from);
  }
}

14Procedural Environment Generation

// how fake-but-realistic AWS topologies are built

When the simulation starts (or you bump the asset-count slider), the environment is rebuilt from scratch. Step 1: spawn N nodes at random positions inside the canvas. Step 2: assign each a service type drawn from a weighted table (more EC2 instances than KMS keys, reflecting real deployments). Step 3: connect nodes whose distance is under a threshold scaled by the edge density slider. Step 4: seed each node's risk attributes from that service type's profile, with small random jitter so no two generations look the same.

Generation Pipeline
function generate(N, density) {
  nodes.length = 0; edges.length = 0;
  for (let i = 0; i < N; i++) {
    const type = weightedPick(SERVICE_WEIGHTS);
    nodes.push({ id: i, x: rand(W), y: rand(H),
      type, ...jitter(PROFILES[type]), compromised: false });
  }
  const reach = 120 * density;
  for (let a = 0; a < N; a++)
    for (let b = a+1; b < N; b++)
      if (dist(nodes[a], nodes[b]) < reach)
        edges.push({ from: a, to: b });
  buildAdjacency();
}

15Threat AI — Pathing Logic

// how a threat decides where to move next

Threats don't wander randomly. Each tick, a threat looks at its current node's neighbors, scores each one, and picks a target weighted toward high asset value and already-weak defenses. The aggression slider sets the probability the threat actually jumps this tick; otherwise it lingers and strengthens its foothold. The patch effectiveness slider reduces compromise probability on each attempt, modeling defenders catching up.

Per-Tick Threat Update
for (const threat of threats) {
  if (Math.random() > aggression) continue; -- wait this tick
  const here = nodes[threat.at];
  const candidates = adj.get(here.id)
    .map(id => nodes[id])
    .filter(n => !n.compromised);
  if (candidates.length === 0) continue;
  const target = weightedPick(candidates,
    n => n.value * 0.6 + n.exposure * 0.4);
  if (Math.random() < (1 - patchEffect)) {
    target.compromised = true; threat.at = target.id;
  }
}
Why value × 0.6 + exposure × 0.4? A real attacker balances what they want (high-value targets) against what they can reach (high-exposure neighbors). Weighting produces pathing that looks intelligent without a full search tree.

16The Rendering Pipeline

// drawing nodes, edges, heat, and animation

Each frame is painted in three layers on a single 2D Canvas. Layer 1: edges as thin lines, darker when dormant and brighter when carrying an active attack. Layer 2: nodes as filled circles; fill color comes from a 256-entry risk-score LUT (same trick as the neural heat map). Layer 3: animated pulses — small dots that travel along compromised edges, their position a simple lerp from source to target driven by a per-pulse progress value.

Frame Render
function render() {
  ctx.clearRect(0, 0, W, H);
  -- 1. edges
  for (const e of edges) drawEdge(nodes[e.from], nodes[e.to]);
  -- 2. nodes
  for (const n of nodes) {
    const color = RISK_LUT[n.score * 255 | 0];
    drawNode(n, color);
  }
  -- 3. attack pulses
  for (const p of pulses) {
    const x = lerp(p.a.x, p.b.x, p.t);
    const y = lerp(p.a.y, p.b.y, p.t);
    drawPulse(x, y);
    p.t += 0.03;
  }
}
