01What Is a Heat Map?

// encoding data as color

A heat map is a way of visualizing data where individual values are represented as colors. Think of a weather radar: blue means cool, red means hot. The same idea applies here — except instead of temperature, we're mapping the activation level of simulated neurons onto a grid of colored cells.

Heat maps work because humans are very good at spotting color patterns. By translating numbers into a smooth gradient of color, we can see structure, clusters, and movement that would be invisible in a table of raw numbers.

No activation (0.0) Low Medium High Peak (1.0)
Each cell in the heat map is colored according to its activation value. Cool tones mean quiet; hot tones mean the neuron is highly active or has just fired.

02What Is a Neural Network?

// biological inspiration, simplified

In your brain, billions of neurons communicate by sending electrical signals to their neighbors. When a neuron receives enough input, it "fires" — sending its own signal onward. This cascading pattern of firing is how you think, see, and move.

Artificial neural networks (used in machine learning) are loosely inspired by this idea. They arrange mathematical functions into layers, pass data through them, and learn by adjusting connection strengths. But they're far more abstract than real neurons.

This simulation sits somewhere in between: it borrows the firing and spreading concept from biology — not the learning or layers from machine learning. It's a visual toy, not a trained model. Think of it as an animated sketch of how neural activation might look if you could see it.

Signal Propagation (Simplified)
A neuron receives activation from neighbors, accumulates it, and when it crosses a threshold, fires — passing activation onward to its own neighbors.

03How This Simulation Models Neural Firing

// grid-based simplification

Instead of modeling individual neurons with complex connections, this simulation arranges them in a flat N×N grid. Each cell in the grid is one neuron. Each neuron can talk to its immediate neighbors (the cells directly adjacent to it).

Why a grid? Two reasons: performance (a flat grid is very fast to iterate over and render) and visual clarity (the grid maps directly to pixels on screen, making the heat map trivial to draw). A 60×60 grid gives us 3,600 neurons updating dozens of times per second — enough to create rich, organic-looking patterns without slowing down your browser.

04The Neuron: Anatomy of a Single Cell

// four properties that drive the simulation

Every neuron in the grid has four properties. Together, they determine when the neuron fires, how it influences its neighbors, and how quickly it returns to rest.

Anatomy of a Simulated Neuron
Activation
0.0 – 1.0
The current "energy level" of this neuron. Starts at zero, accumulates from stimulation and neighbor firing, decays over time.
Threshold
0.1 – 0.9
The activation level a neuron must exceed to fire. Think of it as a trigger point — lower thresholds mean neurons fire more easily.
Decay Rate
0.01 – 0.5
How much activation is lost each tick if the neuron isn't re-stimulated. Prevents activation from building up infinitely.
Refractory Period
0 – 20 ticks
After firing, the neuron enters a cool-down period where it cannot fire again. Creates rhythm and prevents instant feedback loops.

05Activation, Threshold & Propagation

// when neurons fire and how energy spreads

Each tick of the simulation, every neuron checks: is my activation above the threshold? If yes — and if I'm not in a refractory cool-down — I fire. Firing means I take a fraction of my activation and add it to each of my neighbors.

The propagation strength parameter controls what fraction of activation is passed on. If propagation is high, a single firing neuron creates a strong ripple outward. If low, the wave dies quickly.

You can choose between 4-directional neighbors (up, down, left, right) or 8-directional (adds diagonals). Eight directions produce rounder, more organic wave patterns because activation can flow diagonally.

Pseudocode: Firing & Propagation
for each neuron in grid:
  if neuron.activation > THRESHOLD and neuron.refractory == 0:
    -- this neuron fires!
    for each neighbor in get_neighbors(neuron):
      if neighbor.refractory == 0:
        neighbor.activation += neuron.activation * PROPAGATION_STRENGTH / NEIGHBOR_COUNT
        neighbor.activation = min(neighbor.activation, 1.0)
    -- enter cool-down
    neuron.refractory = REFRACTORY_PERIOD
Why divide by neighbor count? This normalizes the energy output so that switching between 4-dir and 8-dir mode doesn't drastically change wave speed — the same total energy is distributed across more or fewer neighbors.
Activation Propagation
A center cell fires (bright) and spreads activation to its 8 neighbors. The intensity of each neighbor's glow reflects the portion of energy received.

06Decay & The Refractory Period

// how the grid returns to rest

Without decay, activation would only ever increase — the grid would rapidly saturate and become a solid bright block. Decay subtracts a small amount of activation from every neuron every tick, ensuring that unstimulated neurons gradually go dark.

The refractory period adds a second layer of regulation. After a neuron fires, it's temporarily "locked out" — it can't fire again for a set number of ticks, and incoming activation from neighbors is ignored during this window. This creates the distinctive wave-like patterns you see in the simulation: activation rolls forward but can't double back on itself immediately, much like a row of dominoes that need to be reset before they can fall again.

Pseudocode: Decay & Refractory Update
for each neuron in grid:
  -- fade activation
  neuron.activation = neuron.activation - DECAY_RATE
  neuron.activation = max(neuron.activation, 0.0) -- floor at zero

  -- count down refractory timer
  if neuron.refractory > 0:
    neuron.refractory = neuron.refractory - 1
Design choice: Decay is applied after propagation in each tick. This means freshly received activation survives for at least one tick before decaying, giving waves a chance to travel rather than dying immediately at their source.

07How the Heat Map Is Rendered

// turning numbers into neon

The rendering step translates each neuron's activation value (a number from 0.0 to 1.0) into a specific color using a color lookup table (LUT). The LUT is an array of 256 pre-computed RGB colors that span the chosen gradient. To color a neuron, we multiply its activation by 255, round to an integer, and look up the corresponding color.

This is drawn on an HTML5 Canvas element. The simulation writes directly to an ImageData buffer — a flat array of pixel values — then paints it to the screen in a single call. This is much faster than drawing individual shapes for each cell, especially at 60×60 resolution or higher.

On top of the base heat map, a second glow pass is drawn for cells with activation above 0.5. This pass uses the Canvas blur filter and additive blending to create the neon bloom effect that makes hot cells appear to radiate light.

08Walking Through One Full Tick

// step by step, start to finish

Every time the simulation advances by one "tick," five distinct phases run in order. Understanding this sequence is the key to understanding everything the simulation does.

Pseudocode: One Complete Tick
-- Phase 1: Random stimulation
repeat SEED_RATE times:
  pick a random neuron
  if it is not in refractory:
    neuron.activation += SEED_INTENSITY

-- Phase 2: Firing and propagation
for each neuron where activation > THRESHOLD and refractory == 0:
  spread activation to neighbors (see Section 5)
  set neuron.refractory = REFRACTORY_PERIOD

-- Phase 3: Decay
for each neuron:
  neuron.activation -= DECAY_RATE
  clamp to [0.0, 1.0]

-- Phase 4: Refractory countdown
for each neuron where refractory > 0:
  neuron.refractory -= 1

-- Phase 5: Render
for each neuron:
  color = lookup_color(neuron.activation)
  draw cell to canvas
apply glow pass for bright cells
Order matters: Seeds are injected first so they can immediately participate in the propagation phase of the same tick. Decay comes after propagation so newly received activation isn't instantly erased.

09What the Parameters Actually Do

// a reference for every slider

The controls panel exposes every tunable parameter. Here's what each one does and what happens when you push it to extremes.

Parameter Controls Low Extreme High Extreme
Grid Size Number of neurons per side (N×N) 20×20 = 400 neurons. Coarse, blocky patterns. 120×120 = 14,400 neurons. Fine detail, may reduce frame rate on slower hardware.
Neighbor Mode 4-dir (cardinal) vs 8-dir (includes diagonals) 4-dir: diamond-shaped propagation patterns, sharper edges. 8-dir: circular wave fronts, smoother and more organic.
Fire Threshold Activation level required for a neuron to fire 0.1: nearly everything fires. Dense, chaotic activation. 0.9: only strongly stimulated neurons fire. Sparse, isolated bursts.
Propagation Strength Fraction of activation spread to neighbors on fire 0.0: neurons fire but don't affect neighbors. No waves. 1.0: strong cascading waves that can travel across the entire grid.
Decay Rate Activation lost per tick when not re-stimulated 0.01: very slow fade. Long glowing trails. 0.5: activation vanishes almost instantly. Only the brightest cells are visible.
Refractory Period Ticks of cool-down after firing 0: neurons can fire every tick. Can cause runaway feedback. 20: long cool-down. Wide "dead zones" trail behind waves.
Seed Rate Random neurons stimulated per tick 1: rare sparks, quiet grid with occasional waves. 50: constant barrage of new activation. Chaotic, noisy.
Seed Intensity How much activation each seed injects 0.1: weak seeds that may not reach the fire threshold alone. 1.0: seeds instantly max out a neuron, guaranteeing a fire.
Glow Intensity Strength of the neon bloom effect 0: no glow, flat colors. 1.0: intense bloom that bleeds across cells, dramatic but can obscure detail.
Tick Speed Simulation ticks per second 1 TPS: slow-motion, study each wave. 60 TPS: real-time fluid animation.

10Real Neural Networks vs. This Simulation

// honest comparison

This simulation captures the spirit of neural activation spreading through a network, but it deliberately omits most of the complexity that makes real neural networks powerful. Here's an honest breakdown.

What this simulation gets right:

• Neurons have activation levels that accumulate from inputs
• Firing is threshold-based — a neuron must reach a certain level to activate
• Refractory periods prevent instant re-firing (real neurons have this too)
• Activation propagates to neighbors, creating cascading waves
• Decay ensures the system doesn't saturate

What this simulation does NOT model:

• Weighted connections — in real networks, each connection has a different strength that changes during learning. Our grid uses uniform weights.
• Backpropagation & training — the process by which neural networks learn. Our grid doesn't learn anything.
• Layers & architecture — real networks have input layers, hidden layers, and output layers. Our grid is a single flat sheet.
• Inhibitory neurons — real brains have neurons that suppress their neighbors. Our neurons only excite.
• Synaptic plasticity — real connections strengthen or weaken with use. Ours are fixed.
• Chemical neurotransmitters — real signaling involves complex chemistry, not simple number addition.

Think of this as an artistic impression of how neural activation flows — useful for building intuition about propagation, thresholds, and refractory behavior, but not a model you'd use for actual neuroscience or machine learning.

11Inside the Code — Grid Storage

// how the program represents thousands of neurons efficiently

All N×N neurons are stored in flat typed arrays rather than nested objects. A Float32Array of length N² holds activation values; a parallel Int8Array tracks each neuron's refractory countdown. Index math replaces per-neuron objects — no allocation churn, no garbage-collection pauses, and the whole grid sits in contiguous memory the CPU can stream through cache.

Grid Allocation & Indexing
const activation = new Float32Array(N * N);
const refractory = new Int8Array(N * N);
const idx = (r, c) => r * N + c;

-- read / write a cell
let v = activation[idx(r, c)];
activation[idx(r, c)] = Math.min(v + delta, 1.0);
Why typed arrays? Plain JS arrays store values as tagged 64-bit references. A Float32Array is raw contiguous memory — 4× smaller per cell, no pointer chasing, and friendly to SIMD-style browser optimizations.

12The Color Lookup Table

// pre-computed colors, zero per-frame math

Mapping activation → color every frame for thousands of cells would be expensive. Instead, a 256-entry lookup table (LUT) is built once at startup by interpolating between the chosen gradient's color stops. Each tick, a neuron's color is one array index away: LUT[activation * 255 | 0]. No multiplications, no branches, no trig in the hot path. Switching color schemes rebuilds the LUT; the render code is untouched.

LUT Construction
const LUT = new Uint8ClampedArray(256 * 4); -- RGBA × 256
for (let i = 0; i < 256; i++) {
  const t = i / 255;
  const [r, g, b] = interpolateStops(gradient, t);
  LUT[i*4 ] = r;
  LUT[i*4+1] = g;
  LUT[i*4+2] = b;
  LUT[i*4+3] = 255;
}

13Canvas Rendering Pipeline

// writing thousands of cells in one call

The heat map is painted by writing pixels directly into an ImageData buffer — a flat Uint8ClampedArray of RGBA bytes — and committing the whole thing to the canvas in a single putImageData() call. One round-trip, no per-cell fillRect overhead. The glow pass is a second render through ctx.filter = 'blur(Npx)' with additive blending, which the browser GPU-accelerates.

Paint Step
const img = ctx.createImageData(W, H);
for (let i = 0; i < N*N; i++) {
  const v = activation[i] * 255 | 0;
  img.data.set(LUT.subarray(v*4, v*4+4), pixelOffset(i));
}
ctx.putImageData(img, 0, 0);

-- glow pass
ctx.filter = 'blur(8px)';
ctx.globalCompositeOperation = 'lighter';
ctx.drawImage(canvas, 0, 0);

14The Main Animation Loop

// decoupling simulation speed from frame rate

The browser's requestAnimationFrame drives rendering at ~60 fps, but the simulation's tick rate is independent and user-controlled (1–60 TPS). A time accumulator bridges the two: each frame adds elapsed milliseconds; whenever the accumulator exceeds the tick interval, one full simulation step runs and the accumulator decrements. Slowing tick speed doesn't choke the render loop — the canvas repaints smoothly even while physics crawls.

Main Loop
let last = performance.now(), acc = 0;
function frame(now) {
  acc += now - last;
  last = now;
  const interval = 1000 / tickSpeed;
  while (acc >= interval) {
    runOneTick(); -- phases 1–4
    acc -= interval;
  }
  render(); -- phase 5
  requestAnimationFrame(frame);
}
requestAnimationFrame(frame);
